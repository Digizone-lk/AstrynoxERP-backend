## Project Overview

BillFlow is a **multi-tenant SaaS billing platform** built for small-to-medium businesses. Each company registers as an independent **Organisation**. Users within an organisation can manage clients, products, quotations, and invoices тАФ all data is strictly isolated between organisations.

## Architecture Decisions

### 1. httpOnly Cookie Auth (not Bearer tokens)

JWT tokens are stored in httpOnly cookies, not in `Authorization` headers or `localStorage`. This prevents XSS-based token theft. Two cookies are issued:

- `access_token` тАФ 30-minute expiry, used on every request
- `refresh_token` тАФ 7-day expiry, used only at `POST /api/auth/refresh`

Each token carries a `type` field (`"access"` or `"refresh"`). All protected endpoints validate the type to prevent refresh tokens from being used as access tokens.

### 2. Multi-tenancy via `org_id` column

Every table except `organizations` and `users` (which join to `organizations`) carries an `org_id` column. All queries filter by `org_id == current_user.org_id`. The org_id is extracted from the JWT тАФ the client never passes it explicitly.

### 3. Soft Delete for Clients and Products

Clients and products are never physically deleted. `is_active = False` soft-deletes them. This preserves referential integrity on historical invoices and quotations that reference them.

### 4. Strict State Machines for Quotations and Invoices

Business logic is enforced at the API layer, not just the DB. Invalid status transitions return `400 Bad Request`.

**Quotation:** `draft тЖТ sent тЖТ approved тЖТ converted` (or `sent/approved тЖТ rejected`)

**Invoice:** `draft тЖТ sent тЖТ paid` or `sent тЖТ overdue тЖТ paid` or `draft/sent/overdue тЖТ cancelled`

### 5. MAX-based Auto-Numbering

Sequence numbers (`QUO-XXXX`, `INV-XXXX`) are generated by querying `MAX(numeric_suffix)` rather than `COUNT(*)`. COUNT-based numbering produces duplicates if records are deleted. MAX-based numbering is monotonically increasing regardless of gaps.

### 6. UUID Primary Keys

All models use UUID v4 primary keys (`postgresql.UUID(as_uuid=True)`). JWT subjects contain the UUID as a string; these are explicitly converted back to `uuid.UUID` objects before DB queries to ensure compatibility across PostgreSQL and SQLite drivers.

## Migration History

Migrations form a **linear chain** managed by Alembic. The `alembic_version` table stores the current head revision.

| Revision | Description | Status |
|---|---|---|
| `557407e76a9b` | Initial schema тАФ creates all core tables | Active DDL |
| `b2c3d4e5f6a7` | Originally "add_auth_tables" тАФ now a no-op | No-op (fixed) |
| `c3d4e5f6a7b8` | Originally "add_clients_products" тАФ now a no-op | No-op (fixed) |
| `d4e5f6a7b8c9` | Originally "add_quotations_invoices" тАФ now a no-op | No-op (fixed) |
| `e5f6a7b8c9d0` | Originally "add_audit_dashboard" тАФ now a no-op | No-op (fixed) |

**Why migrations bтАУe are no-ops:** The initial migration `557407e76a9b` already created every table. The subsequent four migrations tried to `CREATE TABLE` the same tables again, causing `psycopg2.errors.DuplicateTable` on every container start. Rather than deleting the migrations (which would break the chain for deployments that had already applied them), the upgrade/downgrade bodies were replaced with `pass`.

## Features Implemented

### Auth (`/api/auth`)
- Organisation self-registration (creates org + super_admin user in one call)
- Email/password login with httpOnly cookie issuance
- Silent token refresh via refresh cookie
- Logout (cookie clear)
- `/me` тАФ current user profile with org currency

### Users (`/api/users`)
- List, get, create, update users within the organisation
- Super Admin only; role assignment; soft deactivation

### Clients (`/api/clients`)
- Full CRUD (soft delete via `is_active`)
- Product eligibility endpoint: global products + explicitly assigned client-specific products
- Explicit product assignment management

### Products (`/api/products`)
- Full CRUD (soft delete via `is_active`)
- Global vs. client-specific scoping (`is_global` flag)
- Assigned-clients reverse lookup

### Quotations (`/api/quotations`)
- Create with line items (at least one required)
- Sequential auto-numbering: `QUO-0001`, `QUO-0002`, тАж
- Full lifecycle management: draft тЖТ sent тЖТ approved тЖТ converted (or rejected)
- Edit guard: updates only allowed in `draft` state
- Convert to invoice: creates linked invoice, sets quotation to `converted`
- PDF download

### Invoices (`/api/invoices`)
- Create with line items (at least one required)
- Sequential auto-numbering: `INV-0001`, `INV-0002`, тАж
- Full lifecycle management: draft тЖТ sent тЖТ paid / overdue
- Mark-paid and mark-overdue require `super_admin` or `accountant` role
- Cancel (blocked if already paid)
- PDF download

### Dashboard (`/api/dashboard`)
- KPI stats: revenue, outstanding, overdue counts, client/product counts
- 12-month revenue sparkline data

### Reports (`/api/reports`)
- Period-filtered summary: `this_month`, `this_quarter`, `this_year`, `all`
- Invoice KPIs (revenue, invoiced, outstanding, overdue)
- Invoice and quotation status breakdowns
- Revenue by month (up to 24 months)
- Top 10 clients by revenue

### Audit Log (`/api/audit`)
- Automatic log entries on every mutating action
- Filterable by resource type and action
- Paginated, most recent first

## RBAC Design

Dependencies in `app/dependencies.py` enforce role checks. Each dependency extracts and validates the JWT, loads the user from DB, and checks the role.

| Dependency | Allowed Roles |
|---|---|
| `get_any_authenticated` | All authenticated users |
| `get_sales_or_admin` | `super_admin`, `sales` |
| `get_accountant_or_admin` | `super_admin`, `accountant` |
| `get_admin_only` | `super_admin` |

All dependencies also assert `user.is_active == True` and enforce the JWT `type == "access"` check.

## Auto-Numbering Design

`app/services/numbering.py` generates sequential document numbers. The implementation uses `MAX()` on the numeric suffix:

```python
max_num = (
    db.query(func.max(cast(func.substr(Quotation.quote_number, 5), Integer)))
    .filter(Quotation.org_id == org_id)
    .scalar()
) or 0
return f"QUO-{(max_num + 1):04d}"
```

- `func.substr(quote_number, 5)` extracts the numeric part after `"QUO-"`
- `cast(..., Integer)` converts it for numeric comparison
- `or 0` handles the empty-table case
- Numbers are org-scoped: each organisation has its own independent sequence

## Multi-Tenancy Design

Every request goes through a dependency that extracts `org_id` from the JWT. No endpoint accepts `org_id` as a parameter. All data queries include `.filter(Model.org_id == current_user.org_id)`. Attempting to access another org's resource by guessing a UUID returns `404` (not `403`) to avoid leaking the existence of the resource.
